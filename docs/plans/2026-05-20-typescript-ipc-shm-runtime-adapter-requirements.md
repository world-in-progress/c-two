## Decision

Generated TypeScript owns IPC frame validation, route/contract checks, response allocator ingress, request SHM frame encoding, and lifecycle scheduling through structural `responseShmReader` and `requestShmWriter` hooks. It may expose structural Node/POSIX native-buddy wrapper factories that delegate to an injected backend, but it must not implement c2-mem non-dedicated buddy allocation, mapping, or release in pure TypeScript. Complete Node/POSIX non-dedicated SHM support requires a native addon or equivalent FFI backend that can call POSIX SHM APIs and apply c2-mem-compatible buddy allocator operations.

The existing generated Node/POSIX helper remains intentionally dedicated-only. It can read dedicated response segments, mark `read_done`, create dedicated request segments, and unlink request segments after the generated transport observes a matching server response. It is not a buddy backend and must continue rejecting non-dedicated blocks.

## Existing Generated Boundary

The generated TypeScript boundary already provides these runtime-neutral contracts:

- `C2IpcResponseShmReader.read(block, destination?)` maps or copies response SHM bytes into a caller-provided destination or returns an owned bytes object.
- `C2IpcResponseShmReader.release(block)` releases a valid response SHM block after generated post-read validation or before read when allocator/setup failure prevents copying the payload.
- `C2IpcRequestShmWriter.write(payload)` allocates or creates a request SHM block and copies the payload into it before generated IPC frame encoding.
- `C2IpcRequestShmWriter.release(block)` releases only blocks that the generated transport still owns, such as invalid blocks, failed writes, and dedicated blocks after the matching response proves server consumption.
- `C2IpcRequestShmWriter.markConsumed?(block)` is an optional no-free bookkeeping hook for non-dedicated blocks after the generated transport has observed a server response and Rust server code owns cross-process free authority.
- `createNodePosixDedicatedResponseShmReader(...)` and `createNodePosixDedicatedRequestShmWriter(...)` are dedicated-only helpers over an injected file-system contract for runtimes that expose POSIX SHM segments as files.
- `C2NodePosixNativeBuddyResponseShmBackend` and `C2NodePosixNativeBuddyRequestShmBackend` are structural native/FFI backend contracts for non-dedicated buddy support.
- `createNodePosixNativeBuddyResponseShmReader(...)` and `createNodePosixNativeBuddyRequestShmWriter(...)` wrap those backend contracts into the generic C-Two SHM reader/writer hooks while leaving allocator semantics inside the injected backend.
- `C2MemFfiResponsePoolFactory`, `C2MemFfiRequestPoolBinding`, `createC2MemFfiNativeBuddyResponseShmReader(...)`, and `createC2MemFfiNativeBuddyRequestShmWriter(...)` are generated structural facades for runtimes that expose lower-level c2-mem-ffi pool objects rather than the native-buddy backend contract directly.
- `createC2MemFfiRequestPoolFromSymbols(...)` and `createC2MemFfiResponsePoolFromSymbols(...)` live in the `@c-two/c2-mem-ffi` TypeScript package for runtimes that expose JavaScript-callable wrappers over the public C ABI symbols instead of prebuilt pool objects.
- `createNodeIpcConnect({ net })` is a generated structural connector for Node/POSIX Unix-domain sockets. It wraps an injected `node:net`-compatible module into `C2IpcConnect` without importing Node types into generated artifacts; it is transport plumbing only and does not own SHM allocation or c2-mem release semantics.

These contracts are structural so a future Node addon, Bun/Deno FFI adapter, or test fixture can implement them without making C-Two core import FastDB or a concrete Node-specific package.

## Non-Dedicated Response Reader Requirements

A non-dedicated response backend must implement c2-mem-compatible read-and-free semantics for buddy blocks returned by the Rust server:

- It must open the peer buddy SHM segment selected by the response block `prefix`, `segmentIndex`, and `segmentName`.
- It must validate the advertised segment metadata against the concrete mapping size before reading.
- It must copy exactly `byteLength` bytes from `offset` into the provided destination or an owned result.
- It must not return a view that aliases a mapping after `release(block)` can unmap or mutate that mapping.
- It must implement release as the equivalent of Rust `ServerPoolState::read_and_free(...)` / `MemPool::free_at(...)`, including the c2-mem segment header, lock, bitmap, block size, and offset rules.
- It must treat release as a cross-process allocator operation, not as `close()`, `munmap()`, or file-handle cleanup alone.
- It must surface malformed block metadata, mapping failures, allocator-header incompatibility, and free failures as transport-level errors that do not look like CRM user errors.
- It must define segment-cache and mapping lifetime rules so repeated SHM replies do not leak file descriptors or mappings.

The generated transport already calls `release(block)` after successful read, after post-read byte-length validation failure, and after response allocator failure before any response read occurs. The generated native-buddy wrapper rejects dedicated response blocks before invoking the backend, passes a destination view to `readResponse(...)`, copies any accidentally returned bytes into that destination instead of exposing a backend-owned alias, and calls `releaseResponse(...)` itself if post-read validation fails before the generic transport can schedule release. The generated transport rejects dedicated response offsets before invoking the reader. The backend owns the allocator-specific mechanics inside `readResponse(...)` and `releaseResponse(...)`.

Phase 153 provides the first concrete response-side substrate for this backend shape, with Phase 182 tightening the cleanup contract: `c2-mem-ffi` creates a response pool over a server SHM prefix, lazily opens deterministic non-dedicated buddy segments with the advertised buddy segment size, validates `offset + byteLength` against the mapped data region before copying or release, writes bytes only into caller-owned destination memory, releases valid read or unread blocks exactly once through c2-mem `free_at(...)`, tracks successfully read blocks for pool-destroy leak cleanup, and restores read-tracked keys if release validation or free fails. It rejects dedicated blocks so dedicated `read_done` lifecycle remains on the dedicated helper or a future explicit composite adapter.

## Non-Dedicated Request Writer Requirements

A non-dedicated request backend must implement a c2-mem-compatible client buddy pool:

- It must create or open POSIX SHM segments whose names use the advertised client prefix and Rust-compatible buddy segment naming.
- It must initialize the c2-mem buddy segment header, synchronization primitive, bitmap, and allocator metadata in the same layout expected by Rust peers.
- It must advertise the client prefix and concrete buddy segment names/sizes through `C2IpcRequestShmWriter.prefix` and `.segments` before the generated IPC handshake.
- It must allocate a block large enough for the request payload, copy the payload into the block, and return `dedicated: false` metadata with a u16 segment index, u32 offset, and u32 byte length.
- It must release locally only blocks the generated transport still owns, such as invalid returned blocks or blocks whose buddy frame was not successfully written.
- It must not release a valid non-dedicated block after the generated transport receives a server response for the request. At that point Rust server code owns cross-process free authority for accepted request blocks, including early route-admission and capacity reject cleanup.
- It may expose a no-free consumed hook so local bindings can drop stale release authority and ownership tracking without racing the server-side free.
- It must handle local pool exhaustion deterministically so callers can fall back to generated chunked request bytes only by choosing a transport configuration without a request SHM writer.

Request-side non-dedicated support is therefore an allocator backend implementation, not a file-write helper. The generated native-buddy wrapper validates backend prefix and segment metadata before handshake advertisement, but it does not create the pool. If the backend returns a dedicated block, the wrapper releases that block through `releaseRequest(...)` and fails before the generated transport can write a buddy call frame. The generated transport rejects dedicated request offsets and locally releases such rejected blocks before writing any buddy call frame.

Phase 152 provides the first concrete request-side substrate for this backend shape: generated TypeScript now calls optional `markConsumed(block)` for accepted non-dedicated request blocks, while `c2-mem-ffi` creates a c2-mem `MemPool` with dedicated fallback disabled, validates POSIX SHM prefixes against the Rust prefix limit, pre-creates the configured buddy segments so later writes cannot return unadvertised segment indexes, advertises the same prefix, buddy segment names, and data sizes used by Rust IPC handshakes, writes request payloads into non-dedicated buddy blocks, tracks local writer-owned blocks, exposes local release only for blocks still owned by the writer, and exposes `c2_mem_ffi_request_pool_forget_consumed(...)` so a future binding can drop local release authority after the Rust server has accepted cross-process free authority. It is not a generated TypeScript allocator, not a FastDB dependency, and not a complete Node runtime package by itself.

Phase 154 publishes `core/foundation/c2-mem-ffi/include/c2_mem_ffi.h` as the public C ABI contract for the Rust-owned request and response pool substrate. Bindings should include this header rather than hand-writing status values, opaque pool handles, block layouts, constructor limits, or function prototypes. The header intentionally documents request and response blocks as non-dedicated buddy descriptors where `is_dedicated` must be `0`; dedicated support remains on the dedicated helper or a future explicit composite adapter.

Phase 155 adds the first TypeScript package layer for those bindings under `core/foundation/c2-mem-ffi/bindings/typescript`. The package is intentionally structural: it exports `createC2MemFfiNativeBuddyResponseBackend(...)` and `createC2MemFfiNativeBuddyRequestBackend(...)` over lower-level pool objects, plus `createC2MemFfiRequestPoolFromSymbols(...)` and `createC2MemFfiResponsePoolFromSymbols(...)` for symbol-level FFI wrapper objects. It snapshots request prefix/segment metadata for handshakes, validates non-dedicated response and request block ranges before handing them to the pool, releases invalid writer-returned request blocks locally, destroys request pools when metadata discovery fails, and closes pools idempotently. It does not load a dynamic library, call `shm_open`, or claim real Node/Bun/Deno IPC parity by itself.

Phase 156 adds the generated-contract composition layer, tightened by Phase 182: generated TypeScript now exports `createC2MemFfiNativeBuddyResponseShmReader(...)` for response pool factories keyed by server prefix and `createC2MemFfiNativeBuddyRequestShmWriter(...)` for request pools keyed by client prefix. The response helper uses the advertised server buddy segment size unless an explicit `segmentSize` is provided, opens the prefix pool for read or release so allocator-failure cleanup can free unread response blocks, and exposes terminal `close()` to close cached pools and reject later calls. The request helper snapshots prefix/segments through the existing native-buddy writer validation and maps accepted non-dedicated ownership transfer to `forgetConsumed(...)`. This still is not a native loader: application/runtime code must supply the concrete C ABI binding that owns native pointers and POSIX mappings.

Phase 157 chooses the smallest concrete runtime loader that is verifiable in the current repository without introducing a third-party FFI dependency: `@c-two/c2-mem-ffi` ships a Node-API addon source plus a local build script that compiles against the active Node headers and POSIX `dlopen`/`dlsym`. The addon exports a loader for a caller-supplied `c2_mem_ffi` shared-library path and returns JavaScript-callable request/response symbol tables matching the Phase 155 `createC2MemFfiRequestPoolFromSymbols(...)` and `createC2MemFfiResponsePoolFromSymbols(...)` contracts. It owns native pointer handles, explicit destroy/finalizer behavior, C string length/copy out-parameters, block layout conversion, ABI version checks, JS safe-integer-to-u32/u16 guardrails, status propagation, and Uint8Array buffer access; it does not own IPC frames, route validation, FastDB schema/codecs, provider-owned response allocators, or generated contract code. This staged boundary is intentional because Node has no built-in stable FFI in the standard runtime, while Bun/Deno are not available in the current verification environment.

Phase 158 closes the first real Node/POSIX direct IPC traffic gap across both connector and c2-mem-ffi composition layers: generated TypeScript now exposes `createNodeIpcConnect({ net })`, validates the structural socket shape, waits for the Node `connect` event, buffers `data` chunks to satisfy `readExactly(...)`, keeps remote-close EOF distinct from explicit client `close()`, and writes frames through the injected socket; `@c-two/c2-mem-ffi` also exposes a Node UDS convenience connector for package consumers that do not need injected socket modules. The CLI smoke uses a real Unix-domain socket server created with `node:net`, validates the client handshake frame and `FLAG_CALL_V2` route/method/payload frame, and sends an inline reply through the generated transport. The grid FastDB TypeScript smoke then composes the generated direct IPC transport with the c2-mem-ffi Node loader against a real Python/Rust server, proving request SHM, response SHM, allocator ingress, and held responses on the first concrete cross-language path.

## Dedicated Segment Requirements

Dedicated segment support is intentionally narrower:

- Response dedicated blocks have payload bytes after `C2_IPC_DEDICATED_SHM_HEADER_BYTES` and require offset `0`.
- Dedicated response release marks `read_done` so Rust-side dedicated lifecycle can observe completion.
- Dedicated request writers create header-plus-payload segments and return `dedicated: true` with offset `0`.
- Dedicated request release remains client-owned and may unlink the segment only after the generated transport observes the matching server response.
- Dedicated helpers must continue rejecting non-dedicated blocks so users cannot accidentally treat buddy allocator state as a plain file.

## FastDB Boundary

FastDB remains a payload codec and provider-owned response allocator. It may supply `createFastdbC2ResponsePayloadAllocator(module)` and typed facades over generated C-Two clients, but it must not own IPC SHM allocation, buddy free authority, relay route selection, or C-Two frame semantics. A future FastDB-facing TypeScript package may compose the native C-Two SHM adapter with FastDB allocators, but the adapter itself belongs to the C-Two runtime/SDK layer.

## Packaging Direction

The concrete non-dedicated Node/POSIX backend should live outside the generated contract output. Viable packaging shapes are:

- A C-Two TypeScript runtime support package backed by a Rust/N-API native module that reuses c2-mem logic or a bit-for-bit compatible allocator implementation and implements the generated native-buddy backend contracts.
- A thin FFI adapter that exposes `open`, `map`, `alloc`, `free_at`, and lifecycle operations while generated TypeScript keeps the existing structural hook surface.
- Test-only fake adapters that exercise generated frame scheduling but do not claim non-dedicated c2-mem compatibility.

Native or FFI packages must treat `c2_mem_ffi.h` as the source of truth for the current C ABI surface. The header exposes the Rust-compatible prefix and segment-count limits, request/response block layout, status values, and pool function signatures; it does not promise dedicated block support.

Generated contract artifacts may import or receive such backends from application code later, but codegen must remain deterministic, FastDB-neutral, and honest about unsupported runtimes.

For the current Node/POSIX slice, the concrete loader belongs in `core/foundation/c2-mem-ffi/bindings/typescript`, not in generated contract output and not in FastDB. Its build must avoid hidden package-manager side effects: the native addon can be built from checked-in C source using the active Node installation headers and can load either the package-bundled Rust `cdylib` through `loadBundledC2MemFfiNodeNativeSymbols()` or an explicit external Rust `cdylib` path supplied by the caller or test harness. The local package now has an explicit `files` whitelist plus `npm run pack:check` so the dry-run tarball contains the built runtime JS, declarations, Node addon, platform shared library, README, and package manifest only, and so a clean temporary ESM consumer can install the tarball, typecheck against the public declarations, load the bundled library through the public ESM package surface, create real request and response pools, write/read/release small non-dedicated buddy blocks, destroy native request and response handles, and verify closed handles cannot be reused or destroyed twice. The checker may run TypeScript through `TSC_JS` and the current Node executable, or through a direct `TSC` override, and the runtime smoke uses the current Node executable so the package guardrail does not depend on ambient `node` or `tsc` discovery in restricted PATH environments. If a published runtime later wants prebuilds, source-build install scripts, platform-split packages, or a Rust N-API package, that packaging can wrap the same C ABI surface without changing generated artifacts.

The current packed-tarball smoke rejects use-after-destroy and double-destroy calls on raw native request-pool and response-pool handles after their destroy calls, so installed package consumers exercise the Node addon closed-handle guardrail through the public package surface.

## Acceptance Criteria For Broader Completion

The Phase 158/160/161/162/163/165/167/172/174/175/176/182/186/195/196/199/202 grid and generated transport smokes satisfy the first generated direct IPC real-traffic proof for the Node/POSIX c2-mem-ffi loader, including request SHM, response SHM, FastDB allocator ingress, no-allocator owned-byte response reads, materialized output-only calls, held output-only responses, held input/output responses, held raw-buffer invalidation, real-SHM post-read mismatch and release-failure error boundaries, stale-route route-not-found request-SHM cleanup semantics, admitted CRM-error request-SHM cleanup semantics, successful-response request bookkeeping failure semantics, cleanup-failure priority over prior CRM/route errors, response-SHM cleanup when allocator ingress fails before read, generated helper coverage for unread response release through a newly created prefix pool, and non-empty request fallback without `requestShmWriter` while response SHM remains active in materialized no-allocator, materialized allocator-backed, held no-allocator, and held allocator-backed response modes. Before the concrete non-dedicated buddy backend can be marked complete as a reusable runtime package, the implementation must keep that proof and broaden it across the remaining matrix:

- Generated TypeScript exposes a c2-mem-ffi response facade that composes response-side pool bindings with `C2IpcResponseShmReader` hooks.
- A concrete TypeScript runtime binding loads the response-side `c2-mem-ffi` C ABI and verifies the generated facade against real Rust IPC traffic.
- Concrete native/FFI bindings include the public `c2_mem_ffi.h` contract and do not duplicate block layout or status constants by hand.
- The Node/POSIX loader translates C ABI request/response pool symbols into the Phase 155 JavaScript-callable symbol tables with native pointer handles that cannot be double-destroyed.
- Response-side non-dedicated SHM reads work against real Rust server buddy replies and free the server pool block exactly once.
- Response allocator ingress still works when `responsePayloadAllocator` supplies the destination view, and allocator failure before read still releases the real response block.
- No-allocator reads return owned bytes that remain valid after `release(block)`.
- Reader failures, post-read byte-length mismatches, and release failures preserve the existing generated transport error boundary.
- Generated TypeScript exposes a c2-mem-ffi request facade that composes request-side pool bindings with `C2IpcRequestShmWriter` hooks.
- A concrete TypeScript runtime binding loads the request-side `c2-mem-ffi` C ABI and verifies the generated facade against real Rust IPC traffic.
- Generated TypeScript exposes a Node/POSIX UDS connector and verifies generated IPC frame traffic over a real Unix-domain socket, and the grid FastDB smoke verifies the same generated IPC path against a real Rust server with c2-mem-ffi request/response SHM in allocator-backed and no-allocator response modes for materialized and held input/output and output-only methods, plus non-empty request fallback without a request SHM writer in materialized no-allocator, materialized allocator-backed, held no-allocator, and held allocator-backed response modes.
- Request-side non-dedicated SHM writes advertise client buddy segments in the IPC handshake and send Rust-compatible `FLAG_CALL_V2 | FLAG_BUDDY` frames.
- Rust server callbacks can consume request blocks without materializing them into inline bytes first, with direct c2-server coverage proving well-formed buddy request dispatch reaches callbacks as `RequestData::Shm`.
- Accepted non-dedicated request blocks are freed by Rust server authority, while unsent or invalid blocks are released by the local writer.
- Direct IPC remains relay-independent and FastDB-neutral.
- The generated dedicated-only helper still rejects non-dedicated blocks.
- Verification includes Rust c2-mem/c2-server tests, generated TypeScript structure tests, generated TypeScript runtime smoke, C-Two/FastDB bridge smoke, FastDB TypeScript runtime tests, and `git diff --check` in both repositories.

## Non-Goals

Do not add a pure TypeScript non-dedicated buddy allocator that approximates c2-mem from file reads and writes.

Do not move FastDB schema or call-db layout knowledge into C-Two core or generated transport code.

Do not make request SHM mandatory. Chunked request bytes remain the portable fallback when a runtime has no compatible request SHM writer.

Do not release accepted non-dedicated request blocks from the generated TypeScript client after a server response; that would race or double-free against Rust server authority.
